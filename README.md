# Tennis Scoreboard

Веб-приложение для ведения счета теннисных матчей, записи результатов и просмотра истории игр. Проект реализует архитектуру MVCS с использованием серверного рендеринга (SSR).

## Описание

Проект разработан в рамках учебного курса [Java Backend Learning Course](https://zhukovsd.github.io/java-backend-learning-course/projects/tennis-scoreboard/).
Приложение позволяет создавать новые матчи, вести точный подсчет очков по правилам большого тенниса (включая стадии Deuce, Advantage и Tie-break) и просматривать завершенные игры с возможностью фильтрации и пагинации.

## Основной функционал

- **Точная теннисная логика** Подсчет очков (15, 30, 40, Ровно, Больше), геймов и сетов, автоматический переход в тай-брейк.
- **Управление состоянием (In-Memory)** Текущие (незавершенные) матчи хранятся в оперативной памяти (ConcurrentHashMap).
- **Фоновая сборка мусора** Реализован планировщик `ScheduledExecutorService`, который автоматически удаляет брошенные матчи по истечении времени тайм-аута.
- **История матчей** Сохранение завершенных игр в реляционную базу данных (H2) с использованием ORM (Hibernate).
- **Пагинация и Фильтрация** Постраничный вывод истории матчей с возможностью **_поиска по части имени_** любого из игроков.
- **Многослойная архитектура** View -> Controller -> Service -> Repository.
- **Ручное внедрение зависимостей** через `ServletContextListener`.
- **Единая обработка ошибок** реализована через `BaseServlet`.
- **Автоматические миграции БД** при старте сервера с помощью Flyway.
- **Маппинг объектов** с помощью MapStruct.
- **Логирование** с помощью SLF4J + logback.
- **Юнит-тестирование:** покрыты все основные сценарии посчета очков матча, включая тай-брейки.

## Особенности реализации
В проекте постарался соблюсти принципы Clean Architecture:

- **Rich Domain Model:**  Проект избегает антипаттерна Anemic Domain Model. Вся сложная бизнес-логика подсчета теннисных очков инкапсулирована внутри самих доменных моделей (`MatchScore`, `PlayerScore`, `Point`), а не "размазана" по сервисам.
- **Паттерн PRG (Post-Redirect-Get):** При начислении очков (`POST /match-score`) сервер производит вычисления и возвращает `HTTP 302 Redirect` на `GET /match-score`. Это предотвращает двойное начисление очков при обновлении страницы пользователем.
- **Слой UI (JSP & JSTL):** Представление отделено от бизнес-логики. Данные передаются на UI исключительно через DTO-объекты.
- **Двойной репозиторий:** Проект использует два типа хранилищ: `InMemoryOngoingMatchRepository` для быстрых транзакций во время игры и `HibernateMatchRepository` для персистентного хранения.
- **Единая точка обработки ошибок:** Создан абстрактный `BaseServlet`, который оборачивает вызовы `service()` в блок `try-catch`. Он централизованно перехватывает кастомные бизнес-исключения (`MatchNotFoundException`, `InvalidMatchException`) и маршрутизирует их на соответствующие view с правильными HTTP-статусами.
- **Токенизация и динамический поиск (Criteria API):** При фильтрации истории матчей пользовательский ввод не ищется "в лоб". Запрос проходит через `SearchUtil`, где разбивается на токены. Затем с помощью Hibernate Criteria API динамически строится SQL-запрос, который проверяет совпадение токенов с именами игроков.
- **Transaction Runner:** Управление транзакциями вынесено в отдельный инфраструктурный компонент `TransactionRunner`. Это позволяет сервисам выполнять лямбда-выражения внутри транзакционного контекста.

## Установка и локальный запуск
Приложение компилируется в `.war` артефакт. **Для работы приложения необходимо задать переменные окружения для подключения к базе данных**.

### ⚠️ Конфигурация (Переменные окружения)
Перед запуском убедитесь, что в вашей среде установлены следующие переменные:
- `TENNIS_DB_URL` — (Обязательно) JDBC URL для базы данных (например, `jdbc:h2:~/tennisDB`).
- `TENNIS_DB_USERNAME` — (Обязательно) Имя пользователя БД.
- `TENNIS_DB_PASSWORD` — (Опционально) Пароль от БД.
- `TENNIS_CLEANUP_PERIOD_MINUTES` — Интервал работы фонового сборщика "забытых" матчей (по умолчанию `30`).
- `TENNIS_STALE_MATCH_LIFETIME_MINUTES` — Время жизни неактивного матча в памяти (по умолчанию `60`).

### Локальный запуск (через Gretty Plugin)
Самый простой способ запустить проект локально без установки Tomcat — использовать встроенный плагин Gretty, настроенный в `build.gradle`:

1. Клонируйте репозиторий:
```bash
git clone https://github.com/XanderGI/TennisScoreboard.git
cd TennisScoreboard
```
2. Настройте переменные окружения(параметры выше) в вашей IDE или терминале.
3. Запустите проект через Gradle:
```bash
./gradlew appRun
```
4. Приложение будет доступно по адресу: `http://localhost:8080/`

### Деплой на сервер(Tomcat)
1. Клонируйте репозиторий:
```bash
git clone https://github.com/XanderGI/TennisScoreboard.git
cd TennisScoreboard
```
2. Соберите `.war` артефакт с помощью Gradle:
      Для Linux/macOS:
      ```bash
      ./gradlew clean build
      ```
      Для Windows:
      ```bash
      gradlew clean build
      ```
3. Скопируйте собранный файл из `build/libs/TennisScoreboard-1.0-SNAPSHOT.war` в папку `webapps/` вашего сервера Tomcat.

4. Запустите Tomcat.

5. Приложение будет доступно по адресу: `http://ip:8080/TennisScoreboard-1.0-SNAPSHOT/`.

## Структура проекта

```
├───main
│   ├───java
│   │   └───io.github.XanderGI
│   │       ├───config     // Загрузка переменных окружения (AppConfig)
│   │       ├───dto        // Объекты передачи данных для JSP
│   │       ├───entity     // Hibernate JPA сущности (Player, Match)
│   │       ├───exception  // бизнес-исключения
│   │       ├───model      // Доменная бизнес-логика (MatchScore, PlayerScore, Point)
│   │       ├───repository // Интерфейсы и реализации репозиториев
│   │       ├───service    // Слой сервисов (Facade, Calculation, Persistence)
│   │       └───servlet    // Контроллеры 
│   ├───resources
│   │   ├───db.migration   // Миграции БД (Flyway)
│   │   └───logback.xml    // Конфигурация логирования
│   └───webapp
│       ├───css            // Статические стили
│       ├───images         // изображения
│       └───WEB-INF        // JSP шаблоны
└───test                   // Unit-тесты логики подсчета очков (JUnit 5 + Mockito)
```

## 🎓 Что я изучил в ходе реализации проекта
- Проектирование сложных доменных моделей на примере теннисного счета.
- Работа с Hibernate ORM: маппинг связей Many-to-One, использование Criteria API для сложного поиска с пагинацией и динамическими фильтрами.
- Решение проблемы конкурентного доступа к данным (Concurrency) с помощью `ConcurrentHashMap` и блоков `synchronized`.
- Управление жизненным циклом фоновых задач в веб-контейнере с использованием `ScheduledExecutorService`.
- Применение паттерна Post-Redirect-Get (PRG) для защиты от повторной отправки форм.
- Реализация собственного `TransactionRunner`. Это позволило безопасно оборачивать вызовы к БД в единую транзакцию (commit/rollback) без boiler-plate кода.
- Настройка Nginx в качестве reverse proxy для перенаправления запросов в Tomcat и кэширования статических файлов (CSS, Images) для ускорения загрузки view компонентов страницы на стороне клиента.
